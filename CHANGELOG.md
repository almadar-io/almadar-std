# @almadar/std

## 13.13.1

### Fixed

- **`behaviors/functions/index.ts` was missing the `std-tabs-layout` and
  `std-stack-layout` factory exports** in v13.13.0 because the regenerator
  was last run with `--only=<single>` before the package was committed,
  which writes a single-export index. The full sweep that ran later was
  not included in the committed tree. Result: `import { stdTabsLayout }
  from '@almadar/std'` returned `undefined` against v13.13.0.
- This release re-runs the full regenerator and ships the complete
  factory barrel (all 4 layout shells exported, 100 behaviors total).
  No source-level behavior changes — purely a packaging fix.

## 13.13.0

### Added

- **Two new layout-shell molecules** complete the layout-shell set
  alongside `std-master-detail-layout` (13.12.0) and
  `std-dashboard-grid-layout` (13.12.0):
  - **`std-tabs-layout`** — generic tab-navigation shell. Composes
    `std-tabs` (tab strip) with two `std-browse` panels. Default tabs
    "All" / "Active"; rebind via call-site config. Each tab's
    `content: "@trait.X"` reference embeds one of the browse panels so
    only the active panel renders.
  - **`std-stack-layout`** — minimal vertical-stack shell. Composes
    `std-browse` + `std-pagination` with title chrome. Drops Search
    and Filter from `std-filtered-list` for the simplest "show me a
    list" prompt.
- All 4 shells follow the same composition pattern: pure structure,
  every entity field reference stays inside the atoms (browse renders
  rows opaquely via `@config.fields`), atom-internal entities
  (`TabsItem`, `PagedItem`) remain unbound, and the entire shell is
  safe to rebind to any consumer entity through the trait references'
  `linkedEntity` at the molecule call site.

## 5.2.0

### Changes

- **Removed `LOLO_BEHAVIOR_METADATA` and `LoloBehaviorMetadata` exports.** The lolo-specific
  behavior metadata snapshot generated by `scripts/build-lolo-behavior-metadata.ts` is no
  longer shipped. The script and its generated output (`behaviors/generated/lolo-behavior-metadata.ts`)
  are deleted. `@almadar/std` is now pure behavior code with zero coupling to the
  `.lolo` documentation format.

### Deprecated

- **TS factory layer in `behaviors/functions/{atoms,molecules,organisms}/std-*.ts` is deprecated.**
  All 137 factory files (and their `*Params` interfaces) now carry `@deprecated` JSDoc
  tags at the file level. The factories still work — they're the canonical authoring
  path for the ts-to-orb converter — but they are no longer a stable public API or a
  documentation surface. External consumers should migrate to importing the registry
  `.orb` files directly OR via `.lolo` `uses` declarations and reference paths
  (`Alias.entity`, `Alias.traits.X`, `Alias.pages.X`) with overrides at the call site.
  No backwards-compat shims are planned.
- The `behaviors/functions/README.md` has been flipped from "active source, not deprecated"
  to a deprecation banner with a migration path.

### Changed

- **`behaviors/exports/{atoms,molecules,organisms}/*.orb` refreshed from canonical Phase F.10
  registry**. The shipped `.orb` snapshot now matches the post-Phase F.10 atoms (e.g., std-modal
  now has the permanent SAVE transition; std-list/std-inventory have the cleaned-up topology).
  Previously the exports/ snapshot was stale relative to `behaviors/registry/` because it
  was a separate manually-maintained directory; this release brings them into sync.

### Dependency bumps

- `@almadar/core` `>=2.13.0` → `>=5.0.0` (matches the major release that drops `LOLO_TYPE_METADATA`).

### Migration

The only consumer of `LOLO_BEHAVIOR_METADATA` was `tools/almadar-lolo-types-sync` (the
generator that produces `docs/signatures.lolo`). The extraction logic has been relocated
INTO that tool, which now walks the registry `.orb` files directly via `getAllBehaviors()`.
No other consumer existed. If you imported `LOLO_BEHAVIOR_METADATA` from `@almadar/std`
directly: pin to `@almadar/std@5.1.0` if you need the legacy export, or read the registry
`.orb` files yourself from `node_modules/@almadar/std/behaviors/exports/`.

External consumers (`@almadar/agent` gates, Storybook scaffolds, anything that scrapes
`*Params` interfaces): migrate to reading the `.orb` registry files. See
`docs/Almadar_Orb_Behaviors.md` and `docs/LOLO_Gaps.md` in the kflow.ai.builder parent
repo for the orbital-as-function model and the migration story.

### Why

Per the LOLO_Gaps.md consolidation, lolo-specific extraction logic should not live
inside `@almadar/std`. The package should ship behaviors, not documentation metadata.
This release closes that abstraction leak in parallel with the matching `@almadar/core`
5.0.0 release. The `behaviors/exports/` refresh ensures consumers shipping the npm
package see the canonical Phase F.10 surface (with permanent SAVE on std-modal etc.)
instead of the stale pre-Phase-F snapshots that 5.1.0 was carrying.
